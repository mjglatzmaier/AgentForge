## Root Cause (Current)

- Output contract requested from LLM is too large (`ResearchDigest` with full paper objects).
- Prompt currently includes full paper objects, increasing input pressure.
- No per-call budget reducer before provider call.
- No structured retry path tied to truncation/parse failure diagnostics.

---

## Goals

1. Eliminate `finish_reason=length`/truncation failures in normal ranked runs.
2. Keep `digest.json` downstream contract unchanged for `render_report`.
3. Keep replay deterministic.
4. Add diagnostics that make truncation root causes explicit.
5. Keep implementation minimal and low-risk.

- No semantic dedupe/reranker redesign.
- No framework additions.

## Proposed Design (Minimal, High-ROI)

### A) Compact LLM Output Contract
Add a synthesis model used only for provider output, e.g.:
- `SynthesisHighlights`:
  - `query: str | None` (optional convenience)
  - `highlights: list[DigestBullet]`

LLM returns highlights only.  
Python assembles final `ResearchDigest` as:
- `query`: `SynthesisHighlights.query` or `config.query` fallback
- `generated_at_utc`: deterministic current run timestamp policy
- `papers`: copied from selected input artifacts (never generated by LLM)
- `highlights`: from LLM response

This removes most output-token pressure immediately.

### B) Input Compression (Always On)
Before prompt assembly, map each selected paper to compact fields:
- `paper_id`
- `title` (bounded chars)
- `published`
- `categories`
- `abstract_snippet` (bounded chars)

Do not include full author lists or full abstracts in prompt.

### C) Deterministic Budget Guardrail
Heuristic estimator: `est_tokens = ceil(prompt_chars / 4)`.  
If over limit:
1. reduce `abstract_snippet_chars`
2. reduce `max_highlights`
3. reduce selected paper count cap (last resort)

All reductions deterministic and recorded in diagnostics.

### D) Overflow Recovery (Single Retry)
On parse/schema failure with truncation signal:
- retry once with stricter limits
- if still failing, fail explicitly and persist diagnostics artifact

No silent fallback.

### E) Stop-Reason Diagnostics
Record truncation signals in `outputs/synthesis_diagnostics.json`:
- `finish_reason` (if available)
- `overflow_detected`
- `retry_count`
- `prompt_chars`
- `est_prompt_tokens`
- applied limits
- parse/validation error summary + safe excerpts
