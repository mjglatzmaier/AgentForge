# ADR-0000: Reposition AgentForge as an Agent OS

- Status: Accepted
- Date: 2026-03-05
- Decision owners: Maintainers

## Context

AgentForge already has strong primitives (typed contracts, manifest-indexed artifacts, deterministic scheduler/control state).  
The project goal is now broader: evolve into an **agent operating system** with:

- a kernel/control plane for execution and policy enforcement,
- a broker for events/commands,
- service connectors that isolate external systems and secrets,
- a workbench (including Lumen) for approvals, observability, and workflow UX.

We need a migration path with low friction and minimal risk to existing users.

## Options Considered

1. **Refactor in-place on current branch**  
   Fastest start, but higher churn and larger mixed diffs.
2. **Clean branch + full rewrite path**  
   Cleaner history, but slower feedback and greater merge overhead.
3. **Side-car implementation in current repo (chosen)**  
   Add Agent OS modules beside existing runtime and migrate incrementally.

## Decision

Adopt a **side-car architecture** inside the current repository and branch flow:

- Keep current orchestrator/runtime operational.
- Introduce Agent OS v1 contracts and components in parallel.
- Add compatibility adapters to run existing agents through the new kernel.
- Promote v1 interfaces only after parity checks and acceptance tests pass.

This is the least-friction path while preserving release safety.

## Architectural Direction (v1)

1. **Kernel / Control Plane**
   - deterministic scheduling and retries,
   - capability + policy decision point,
   - approval checkpoints for irreversible operations,
   - run/event/artifact lineage.
2. **Broker**
   - typed, append-only event bus contract,
   - command/request routing between kernel and connectors,
   - replay-safe IDs and correlation IDs.
3. **Service Connectors**
   - isolated boundary to external APIs/resources,
   - secrets stay in connector boundary + OS keychain/secret manager,
   - strict operation schemas and bounded error taxonomy.
4. **Workbench Clients (Lumen)**
   - read/control APIs for runs/events/approvals/artifacts,
   - no direct tool execution or secret access.

## Consequences

### Positive

- Controlled migration with fewer breaking changes.
- Clear security boundaries and least-privilege enforcement.
- Extensible connector model across platforms and runtimes.

### Trade-offs

- Temporary duplication while side-car and legacy paths coexist.
- Additional compatibility adapters during transition.

## Guardrails

- Keep contracts typed and versioned (`*-v1.md` specs + Pydantic models).
- Deny-by-default capability policy.
- No raw secrets in kernel/agent memory beyond required operation scope.
- Cross-platform transport assumptions (HTTP/Unix socket/Named Pipe abstraction).
- Avoid framework-heavy rewrites; prefer small deterministic modules.
