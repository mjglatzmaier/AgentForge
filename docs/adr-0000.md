# ADR-0000: AgentForge Microkernel Control Plane

## Status
Proposed

## Context
AgentForge is evolving from a pipeline orchestrator into a personal agent runtime that will handle high-trust connectors (Gmail, trading) and multi-agent workflows (news/OSINT/research). This increases risk (prompt injection, accidental tool misuse, secret leakage) and requires:

- least-privilege execution
- auditable runs (why did it act?)
- approval-gated actions (send email, place trades)
- clean separation between “kernel” and “services”

## Decision
Refactor AgentForge into a microkernel control plane:

- **Kernel = small, dumb, policy-heavy**
- **Everything else = user-space services** (connectors, agents, compute)

### Kernel responsibilities (agentd)
1) Run orchestration (run/step state machine)
2) Tool broker (routes tool calls; no business logic)
3) Policy enforcement (capabilities + approvals + rate limits)
4) Event log (append-only) + artifact registry
5) Scheduling/triggers
6) Local IPC/API server (CLI + Lumen)

### User-space services
- Connectors: gmaild, exchanged, rssd, etc.
- Agent runtimes: python agents (and later containerized or multi-language agents)
- Optional compute services: embedding, ranking, summarization (local or remote)

## Rationale: Python is acceptable for safe + cross-platform
Security comes from architecture boundaries rather than kernel language:

- Kernel never exposes raw secrets to agents
- Tools run in separate processes with narrow scopes
- Capability checks happen before any tool call
- Approval gates exist for irreversible actions

Python provides iteration speed and cross-platform support; the security posture is enforced by isolation + policy.

## OS Integration Level (Minimum Required)
Cross-platform baseline (macOS/Windows/Linux):
- Run `agentd` as a background process (service install optional later)
- Store secrets in OS keychain (via `keyring`)
- Use local IPC (HTTP on localhost + WebSocket event stream)
- Restrict filesystem permissions for run directories as feasible

Optional later (not required for MVP):
- systemd unit (Linux)
- launchd plist (macOS)
- Windows service wrapper
- deeper instrumentation (eBPF/ETW)

## Lumen Integration
Lumen is a client “Workbench”, not the runtime:

- Lumen (C) connects to `agentd` via local API (HTTP+WS or gRPC later)
- Lumen renders: pipeline graph, event timeline, tool traces, artifact browser, approval prompts

No secrets in Lumen; secrets remain inside connector services + OS keychain.

## Consequences
**Pros**
- Least-privilege by default
- Auditable and replayable runs
- Safe extensibility: new tools/connectors without widening blast radius
- Great debugging UX when paired with Lumen

**Cons**
- More moving parts (multiple services)
- Requires strict contracts + policy discipline

## Notes
Microkernel litmus test:
> Even if an agent is prompt-injected, the worst it can do is bounded by its capabilities and approval gates.
